# ShareUpdateAuditLoggerSample

.NET Framework 4.8.1 / Windows Forms / Windows 11 向けの勉強用サンプルです。  
FileSystemWatcher でフォルダー配下の変更を検知し、Security イベントログ 4663 を参照して、更新者候補を画面グリッドへ表示します。

## このサンプルで見たいもの

このサンプルは、次の流れを画面で追えるようにしたものです。

1. FileSystemWatcher が変更通知を受ける
2. Security ログの 4663 を検索する
3. 対象ファイルに近いイベントを見つける
4. 最終更新者、ドメイン、プロセス名をグリッドへ出す

大事な点は、**最終更新者は FileSystemWatcher 単体では取れない**ことです。  
更新者は Security ログ 4663 側を読んで補います。

## できること

- ファイルの作成、更新、削除、名前変更を検知
- Security ログの 4663 から次の情報を取得
  - 最終更新者
  - ドメイン
  - プロセス名
  - イベント ID
  - アクセス内容
- 画面グリッドへ一覧表示

## 画面で出す項目

- 検知時刻
- 変更種別
- フルパス
- ファイル名
- 更新時間
- 最終更新者
- ドメイン
- プロセス名
- イベントID
- 備考

## 前提

- Windows 11 上で動かす想定
- 監視対象はローカルパスでも共有フォルダー実体パスでも可
- Security ログの 4663 が出ていない場合、最終更新者は空欄
- Security ログを読むため、アプリの起動権限が重要
- 監査ポリシーだけでは足りず、対象フォルダー側の監査設定も必要

## プロジェクト構成

- MainForm
  - 画面
  - FileSystemWatcher の開始、停止
  - グリッド表示
- Models/FileAuditLogEntry
  - グリッド 1 行分の表示データ
- Services/FileAuditEventLogReader
  - Security ログ参照
  - 4663 候補取得
  - 対象パス一致判定
- Services/EventAuditInfo
  - 監査ログから抜いた値

## 動かす前の準備

ここが一番詰まりやすい部分です。  
順番どおりに進めると切り分けしやすくなります。

### 1. Visual Studio の起動方法

Security ログを読むため、**デバッグ実行時は Visual Studio を管理者権限で起動**します。

手順:

1. Visual Studio を終了
2. Visual Studio を右クリック
3. **管理者として実行** を選択
4. そのまま本プロジェクトを開いてデバッグ実行

通常起動の Visual Studio から実行すると、次のような状態になりやすいです。

- Security ログ参照権限不足
- `UnauthorizedAccessException`
- 最終更新者が空欄

このときはコードより先に起動権限を見直します。

### 2. PowerShell を管理者権限で開く

監査ポリシー設定は、**管理者権限の PowerShell** で実行します。

手順:

1. スタートメニューで `PowerShell` を検索
2. **管理者として実行** を選択
3. 次のコマンドを順に実行

### 3. 監査ポリシーの確認と有効化

この環境で実際に確認できたコマンドをそのまま載せます。

#### 一覧確認用コマンド

```powershell
auditpol /list /category
auditpol /list /subcategory:*
auditpol /list /subcategory:* /v
```

用途:

- 監査カテゴリ一覧を見る
- サブカテゴリ一覧を見る
- GUID 付きで確認する
- 日本語環境で正しい表示名を確認する

#### 最小構成で必要なコマンド

```powershell
auditpol /set /subcategory:"ファイル システム" /success:enable
auditpol /get /subcategory:"ファイル システム"
```

期待する結果:

```text
システム監査ポリシー
カテゴリ/サブカテゴリ                               設定
オブジェクト アクセス
  ファイル システム                               成功
```

#### カテゴリ込みで設定・確認するコマンド

```powershell
auditpol /set /category:"オブジェクト アクセス" /subcategory:"ファイル システム" /success:enable
auditpol /get /category:"オブジェクト アクセス"
```

この環境では上のコマンドも正常終了しました。  
ただし、`/get /category:"オブジェクト アクセス"` を実行すると、**オブジェクト アクセス配下の他サブカテゴリも成功表示になる**形で見えました。  
サンプル用途としては、まず `ファイル システム` だけを有効化して確認し、その後必要に応じて広げる流れの方が追いやすいです。

#### うまくいかなかったコマンド

最初に次のコマンドは失敗しました。

```powershell
auditpol /get /category:"Object Access"
```

結果:

```text
エラー 0x00000057
パラメーターが間違っています。
```

この環境では英語名指定で失敗し、日本語名指定で通りました。  
日本語環境では、先に `auditpol /list /subcategory:*` か `auditpol /list /subcategory:* /v` を実行し、表示名をそのまま使う方が安全です。

### 4. 対象フォルダーへ監査設定を追加

監査ポリシーを有効化しただけでは 4663 は出ません。  
**監視対象フォルダー側の監査設定** が必要です。

手順:

1. 監視対象フォルダーを右クリック
2. **プロパティ**
3. **セキュリティ**
4. **詳細設定**
5. **監査** タブ
6. **追加**
7. プリンシパルを選択
   - 最初の確認では `Everyone` でも可
   - 実運用では対象ユーザーやグループへ絞る
8. 種類は **成功**
9. 適用先は **このフォルダー、サブフォルダーおよびファイル**
10. 次の書き込み系、変更系を入れる

確認しやすい権限例:

- 書き込み
- 追加データ
- 属性の書き込み
- 拡張属性の書き込み
- 削除
- サブフォルダーとファイルの削除

最初は広めに入れて 4663 が出ることを確認し、その後で必要なものへ狭める方が分かりやすいです。

### 5. Event Viewer で 4663 を確認

ファイルを保存したあと、イベント ビューアーで 4663 が出ていることを確認します。

手順:

1. **イベント ビューアー** を開く
2. **Windows ログ** → **セキュリティ**
3. **現在のログをフィルター**
4. イベント ID に `4663` を入力

確認したい項目:

- SubjectUserName
- SubjectDomainName
- ObjectName
- ProcessName
- AccessList

このサンプルでは主に次を使っています。

- SubjectUserName
- SubjectDomainName
- ObjectName
- ProcessName
- AccessList

### 6. 動作確認の順番

最初は名前変更より、**新規作成 → 上書き保存** の順で見る方が追いやすいです。

おすすめ手順:

1. 監視対象フォルダーを空に近い状態へする
2. `test.txt` を作成
3. メモ帳などで 1 行追記して保存
4. Event Viewer の 4663 を確認
5. アプリのグリッドを確認
6. その後で名前変更を試す

この順にしておくと、

- 4663 が出ていないのか
- 4663 はあるがアプリが拾えていないのか
- 名前変更だけがずれているのか

を切り分けしやすくなります。

## 使い方

### 1. アプリ起動

- デバッグ実行時は管理者権限で起動した Visual Studio から実行
- exe 直接実行時も必要に応じて管理者権限で起動

### 2. 監視対象フォルダーを指定

- 参照ボタンで対象フォルダーを選択
- 共有フォルダー実体が同一マシン上にある場合は、その実体パスを指定

### 3. イベントログ検索秒数を設定

初期値は `30` 秒です。  
まずはこの値のままで確認するのが無難です。

考え方:

- 小さすぎると 4663 の反映待ちで取り逃しやすい
- 大きすぎると無関係な候補も増える

まずは `30`、必要に応じて `10` ～ `60` の範囲で調整します。

### 4. 監視開始

監視開始ボタンを押すと、次の流れで動きます。

1. Security ログを読めるか確認
2. FileSystemWatcher を開始
3. 変更通知を受信
4. Security ログ 4663 を検索
5. 対象パスに一致する候補から更新者を採用
6. グリッドへ表示

## 備考列の見かた

- `4663 一致`
  - 対象パスに一致する監査ログを取得
- `監査ログ未検出。監査反映待ち、監査設定、検索秒数を確認してください。`
  - 4663 がまだ見つかっていない
  - 監査設定不足、検索秒数不足、保存直後すぎる、のどれか
- `4663 取得済みだが対象パス不一致。先頭候補: ...`
  - Security ログは読めているが、対象ファイルとの一致判定で外れている
- `名前変更: old -> new`
  - Rename 通知時の表示

## よくある詰まりどころ

### 1. Security ログ参照権限不足

症状:

- 監視開始時に権限不足ダイアログ
- Visual Studio の出力に `UnauthorizedAccessException`
- 最終更新者が空欄

見直し先:

- Visual Studio を管理者として起動しているか
- exe を通常権限で起動していないか

### 2. 4663 が出ない

見直し先:

- `auditpol /get /subcategory:"ファイル システム"` が成功になっているか
- 監視対象フォルダーに監査設定を入れたか
- 実際に変更したファイルがその監査対象配下か

### 3. 4663 は出るが最終更新者が空欄

見直し先:

- アプリを管理者権限で起動しているか
- 検索秒数が短すぎないか
- 対象ファイルと 4663 の ObjectName が一致しているか

### 4. コマンドが通らない

見直し先:

- PowerShell を管理者権限で開いているか
- 英語名ではなく日本語表示名で指定しているか
- 先に `auditpol /list /subcategory:*` で表示名を確認したか

## 実装上の考え方

- FileSystemWatcher は変更通知だけを担当
- 更新者の取得は Security ログ 4663 を参照
- 4663 は複数出ることがあるため、直近候補を広めに取り、その中から対象パス一致で絞る
- Security ログ反映は通知と完全同期ではないため、検索秒数を持たせる

## 補足

- 監査設定を広くしすぎると Security ログが多くなる
- 実運用へ近づける場合は、対象フォルダー、対象ユーザー、監査権限を必要範囲へ絞る方が扱いやすい
- 今回のサンプルは勉強用のため、理解しやすさを優先している
