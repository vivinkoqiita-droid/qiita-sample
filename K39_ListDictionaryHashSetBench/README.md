# K39 List / Dictionary / HashSet 比較ベンチマーク

`List<T>` / `Dictionary<TKey, TValue>` / `HashSet<T>` を、実務で出やすい処理ごとに比較するための WinForms サンプルです。  
API の説明よりも、既存コードの見直し材料として使う想定です。

## このプロジェクトで見られること

このアプリでは、次の 7 パターンを同じ画面から実行できます。

1. 末尾の会員番号を何度も探す
2. 注文番号から明細を引く
3. 重複行を除いて取り込む
4. 退会者IDを除外して残す
5. 2つの名簿で共通IDを探す
6. 更新対象IDを探して書き換える
7. 明細ごとに会員か判定する

結果一覧には次の内容を表示します。

- 開始時刻 / 終了時刻
- `List` / `Dictionary` / `HashSet` の実測時間
- 最速だったコレクション
- 選択中シナリオの補足説明

## 画面の見方

上段で対象件数と反復回数を指定し、全件または選択行だけ実行できます。  
中央の一覧で各シナリオの結果を見て、下段の概要欄で処理の意味と最速比を確認する流れです。

- **対象件数**: 比較対象データの件数
- **反復回数**: 各シナリオの計測回数
- **全件を実行**: 7 シナリオを順に実行
- **選択行を実行**: 選択中シナリオだけ実行
- **結果クリア**: 計測結果を初期化

## 計測の前提

この比較は、各場面で書きやすい実装をそのまま並べています。  
そのため、参照用コレクションを先に作ってから判定だけ測る場面もあれば、追加や更新まで含めて処理全体を測る場面もあります。

見たいのは「常にどれが最速か」ではなく、**この書き方だとどこで差が開くか** です。

初回実行はウォームアップを 1 回入れてから平均を出しています。  
これは JIT コンパイルや初期化の影響を本番の計測から外すためです。

## 実装の見どころ

### MainForm.cs

このファイルにベンチマーク本体があります。

- シナリオ定義の登録
- 一覧表示の更新
- 単体実行 / 全件実行
- 計測処理
- 結果の整形

ベンチマークメソッドは次の 7 本です。

- `BenchmarkContains`
- `BenchmarkRandomLookup`
- `BenchmarkDuplicateAdds`
- `BenchmarkRandomRemove`
- `BenchmarkIntersection`
- `BenchmarkUpdateLookup`
- `BenchmarkMembershipFilter`

### MainForm.Designer.cs

画面部品の配置と初期値です。  
件数入力、反復回数入力、結果一覧、概要欄、実行ボタンを定義しています。

### Program.cs

アプリ起動処理です。  
`ApplicationConfiguration.Initialize();` の後に `MainForm` を開きます。

## 各シナリオの見どころ

### 1. 末尾の会員番号を何度も探す

同じ値に対する存在確認を大量に回す場面です。  
`Contains` をループ内で使っているコードの見直し候補になります。

### 2. 注文番号から明細を引く

ID から 1 件引く場面です。  
画面表示や API 参照の前段に近い比較です。

### 3. 重複行を除いて取り込む

CSV 取込や ID 一覧生成の前段を意識した比較です。  
一意な値だけ残したい場面を見ています。

### 4. 退会者IDを除外して残す

除外一覧を見ながら残す側を作る場面です。  
停止ユーザー除外や対象外コード除外に近い処理です。

### 5. 2つの名簿で共通IDを探す

左右の一覧を突き合わせて、両方にある値だけ拾う場面です。  
マスタと明細の突合で出やすい比較です。

### 6. 更新対象IDを探して書き換える

更新対象 ID を受け取り、対象データを探して値を書き換える場面です。  
一括更新の前段で差が出やすい処理です。

### 7. 明細ごとに会員か判定する

明細を 1 件ずつ見ながら、会員一覧に存在するか判定する場面です。  
集計前の前段や対象抽出に近い比較です。

## 結果を読むときの注意点

- 件数が少ない場合は差が目立ちにくい
- 1 回しか検索しない処理では、事前変換コストの見え方が変わる
- 順番そのものが重要なら `List<T>` が自然な場面もある
- 実データでは comparer やオブジェクト構造の影響も出る

そのため、このアプリは「絶対にこれを選ぶ」という結論用ではなく、既存コードの見直し起点として使うのが向いています。

## 触る箇所の目安

### シナリオを追加したい場合

`InitializeScenarios` に定義を追加し、対応する `Benchmark...` メソッドを `MainForm.cs` に増やします。

### 列や表示文言を変えたい場合

- 一覧列: `InitializeGrid`
- 下段の表示内容: `UpdateOverview`
- 実行結果の反映: `ApplyStateToGrid`

### 件数や回数の既定値を変えたい場合

`MainForm.Designer.cs` 内の `NumericUpDown` 初期値を変更します。

## 使いどころ

次のようなコードを見直すときに使いやすいサンプルです。

- `Contains` がループ内にある
- `FirstOrDefault(x => x.Id == id)` を何度も呼んでいる
- 追加前の重複確認を毎回書いている
- 除外一覧が `List<T>` のまま残っている
- 更新対象ごとに一覧を探している
